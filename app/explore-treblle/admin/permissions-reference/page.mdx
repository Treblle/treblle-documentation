# Permissions Reference

<NoteCard>
Quick reference guide for all permissions in Treblle. Use this when creating custom roles or auditing access levels.
</NoteCard>

## Overview

Treblle's permission system provides granular control organized by:
- **Feature Area** - What the permission controls
- **Plan Level** - Free, Starter, Team, or Enterprise
- **Role Type** - Who typically needs this permission

## Permission Matrix by Role

### Administrator Role
**ALL PERMISSIONS** across all features and plans. For platform administrators only.

### Owner Role (Default: 33 permissions)

**Full workspace control including:**

| Category | Permissions |
|----------|-------------|
| Workspace | All settings, billing, SSO, API creation/deletion |
| User Management | Invite, edit, remove users; manage roles and groups |
| API Access | Full access to all APIs and features |
| Documentation | View and edit all documentation |
| Analytics | All dashboard widgets (plan-dependent) |
| Compliance | Full governance and compliance (plan-dependent) |

<CautionCard>
Owner role cannot be deleted as it's a system role. At least one Owner must exist in each workspace.
</CautionCard>

### Member Role (Default: 14 permissions)

**Standard team access:**

| Category | Permissions |
|----------|-------------|
| Requests | View, search, filter, comment |
| API Docs | View documentation |
| Dashboard | Basic widgets |
| Workspace | No settings access |
| Billing | No billing access |
| User Mgmt | Limited (plan-dependent) |

<TipCard>
Most development team members should have the Member role with API access granted separately.
</TipCard>

### Customer Role (Default: 1 permission)

**External user access:**

| Category | Permissions |
|----------|-------------|
| Requests | View own requests only |
| API Docs | View assigned API docs |
| Everything Else | Read-only, no admin access |

**Typical use:** Clients, partners, external auditors

### ⚫ No Access Role

**Explicitly deny permissions** - Used for granular control when temporarily restricting access.

## Creating Custom Roles

1. **Start with a template** - Base on Owner or Member, then customize
2. **Least privilege** - Only grant necessary permissions
3. **Check plan limits** - Some permissions require specific plans
4. **Name clearly** - Use job titles (e.g., "Frontend Developer", "QA Engineer")
5. **Document purpose** - Keep notes on why each custom role exists
6. **Test first** - Assign to test user before rolling out
